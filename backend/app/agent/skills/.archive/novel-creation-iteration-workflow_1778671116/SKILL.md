## 工作流程

### 第一步：需求采集
与用户沟通，确认以下信息：
- **小说类型**：科幻/悬疑/都市/奇幻/二次元等
- **核心风格**：爽感/虐心/烧脑/热血等
- **篇幅预期**：短篇/中篇/长篇连载
- **特殊偏好**：如二次元、反差萌、系统流等
- 如果用户给出明确偏好，直接记录并跳过通用建议
- 更新用户偏好信息至 memory-get / user-profile-update

### 第二步：创意发散（多方案）
根据用户偏好，一次性提供 3-4 个差异化的大纲选项，每个方案包含：
- **设定描述**：一句话概括核心创意
- **核心爽点说明**：为什么要读这个故事
- **简要的世界观或背景**：基本的故事框架
- **潜在难点提示**：让用户有预期
- **预估创作难度**：如「这个设定构思量大」
- 方案之间要有明显差异（题材/设定/叙事角度）

### 第三步：收集专家级用户反馈
主动邀请用户点评每个方案的优缺点，引导用户从以下维度评估：
- **创作难度**：是否需要极大的世界构思/缜密逻辑/复杂计算
- **阅读体验**：读者是否会感到吃力/丧失快乐感
- **场景转化**：如何避免困在单一场景/副本
- **爽点设计**：爽点是否清晰、是否能持续给读者带来满足感

记录用户在以下维度的评价：
- 创作难度（设定复杂度、逻辑要求）
- 阅读体验（是否烧脑、是否轻松）
- 场景转换（是否单一、是否多元）
- 核心方向认可度

### 第四步：提炼创作原则
根据用户反馈，总结新的创作约束条件：
- 设定简单但展开丰富
- 阅读轻松不烧脑
- 场景转化自然
- 爽点密集
保留验证通过的优秀方向，放弃创作难度过高的方案

### 第五步：重新生成优化方案
应用新原则，生成 2-3 个改进后的方案，明确标注改进点：
- 解决原有问题
- 保留有效元素
- 包含爽点设计和场景推进方向
- 明确说明改进了什么（呼应用户反馈）
- 被否定的方案说明问题所在，但不在此基础上修补，而是重新设计

### 第六步：迭代筛选
重复「用户点评→方案优化」循环，直到用户确定最终选择：
- 每次迭代时明确告知用户改进点，列出具体改进了什么
- 被否定的方案说明问题所在，但不在此基础上修补，而是重新设计
- 保留被认可的方案核心，淘汰问题方案
- 保留可行方向，放弃高门槛方向

### 第七步：结构化深度拆解
针对选定方案，从以下维度展开：
- **主角人设**：身份/性格/现状/目标/金手指
- **核心元素定位**：如宠物/系统/职业的表面vs真实
- **世界观**：背景/等级体系/核心矛盾
- **剧情大纲**：按篇章拆分，标注核心爽点
- **爽点节奏表**：按阶段规划打脸/逆袭/觉醒时机

### 第八步：用户确认具体选择
提供多维度选项让用户锁定细节：
- 主角性格类型
- 核心元素性格类型
- 开局场景类型
- 通过多选项让用户快速确定关键要素
- 逐层拆解确认：主角人设、元素性格、开局场景等
- 每组选项不超过 3 个，避免选择困难
- 确认后再展开完整大纲

### 第九步：正文撰写
基于以上所有决策，撰写正文内容，覆盖：
- 开篇章节（至少 3 章）
- 已埋伏笔（后续剧情线索）
- 明确的章节结尾钩子
- 结构化输出：世界观设定 → 主角人设 → 核心元素定位 → 剧情大纲 → 爽点节奏表
- 邀请用户进一步调整

### 第十步：迭代续写
根据用户反馈决定：
- 直接续写后续章节
- 调整已写内容
- 展开新篇章

---

### 关键原则
- 每次迭代都要明确告知用户改进内容，并列出具体改进了什么
- 被否定的方案要说明问题所在，但不在此基础上修补，而是重新设计
- 最终引导用户确定一个方向后，进入详细构思阶段
- 保留被认可的方案核心，淘汰问题方案
- 询问用户对各方案的评价，引导其指出创作难点或逻辑问题
- 对专业反馈表示认可并学习，持续优化创作约束条件
- 迭代循环直到用户满意再进入下一阶段

### 适用场景
小说创作、故事大纲设计、游戏设定、创意选题等需要迭代优化的场景
