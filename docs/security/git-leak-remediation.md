# Git 泄露处置记录（基础设施信息）

## 事件分级

- 传播范围：已推送到私有仓库。
- 泄露内容：主机别名、域名、部署路径等基础设施元数据。
- 密钥状态：未发现可直接利用的密钥/口令/令牌。

结论：按中风险信息泄露处理，优先采用“新增降敏提交 + 平台侧清理”，暂不强制历史重写。

## 已完成止血

- 已将部署文档与示例配置替换为占位符，不再在仓库暴露真实主机/域名/绝对路径。
- 已将部署脚本改为参数化输入，避免默认输出真实生产地址。

## 远程平台清理清单

> 当前环境未安装 `gh` CLI，请在 GitHub Web 控制台逐项完成。

1. 在仓库 Code Search 搜索以下关键字，确认最新分支已不可检索：
   - `<LEAKED_DOMAIN>`
   - `<LEAKED_DEPLOY_DIR>`
2. 在最近 PR 的 `Files changed`、Review Comment、Issue Comment 中搜索同样关键字，编辑或删除包含泄露内容的评论。
3. 检查发布说明（Release Notes）和 Wiki 页面是否引用旧提交中的敏感信息。
4. 如组织开启代码索引缓存，向管理员申请重新索引/缓存刷新。
5. 将此次事件记录到团队安全日志，标记“仅元数据泄露，无凭据泄露”。

## 是否需要历史重写（决策）

当前决策：**不执行历史重写**（保留审计连续性，降低协作扰动）。

理由：
- 仓库为私有，访问面可控；
- 未泄露直接可利用凭据；
- 已通过新增提交完成快速降敏。

## 升级为历史重写的触发条件

若出现任一条件，立即升级到 Path B（重写历史）：

- 发现泄露内容被转发到公开渠道；
- 发现仓库成员范围超出最小权限并无法快速收敛；
- 发现同一提交中存在密钥、令牌、私钥、数据库口令；
- 合规要求“历史中不可见”。

## Path B 参考命令（仅触发时执行）

```bash
# 1) 回退并重新提交（仅处理最新一次提交）
git reset --soft HEAD~1
# 编辑并降敏后重新提交
git commit -m "docs: redact production infra metadata"
git push --force-with-lease

# 2) 多提交清洗（示例，需先安装 git-filter-repo）
git filter-repo --path docs/deploy.md --invert-paths
# 或使用 --replace-text 做定向替换
```
